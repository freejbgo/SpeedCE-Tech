## 写在前面：出海网站最怕的，不是「慢」，而是「不知道谁慢」

做全球化业务的人，几乎都经历过这种撕裂感：

- 你在上海办公室打开 `.com`，秒开，觉得上线成功；
- 销售说德国客户反馈「页面转圈」；
- 运维 ping 了一下美国办公室，延迟 180ms，说「正常啊」；
- 老板问：「我们到底能不能服务欧美？」——没人能答上来。

问题不是团队不专业，而是 **测速视角错了**。

你在本地测一次，只能代表你；你在服务器上 `curl localhost`，只能代表机器自己。出海业务需要的是：

> **从目标市场所在地的远端节点，向你的域名发起真实 HTTPS 访问，把结果画在地图上。**

本文是一份 **网站出海全链路测速验收手册**：从机房选址、CDN 策略、DNS 解析到上线后巡检，说明什么时候看中国节点、什么时候看全球节点、地图怎么读、不同市场的达标线是什么。全文以 SpeedCE 为操作示例——它在一页内提供 **中国节点地图 + 全球节点地图** 双视图，特别适合出海团队。

## 第一章：出海测速的三个基本问题

动手之前，先回答这三个问题，否则测了也白测。

### 1.1 你的用户在哪？

| 业务类型 | 优先看的地图 | 次要参考 |
|----------|--------------|----------|
| 纯海外（欧美 SaaS） | 全球节点 | 中国节点（团队管理用） |
| 出海 + 国内团队 | 两者都要 | — |
| 东南亚电商 | 全球节点（新、马、泰、印尼） | 中国节点 |
| 中东 / 拉美新兴 | 全球节点对应国家 | — |

**没有「全球通吃」的单次测试**——你要测的是 **目标市场**，不是「全球平均」。

### 1.2 你测的是域名还是 IP？

| 测什么 | 适用场景 |
|--------|----------|
| 域名（HTTPS） | 上线验收、CDN 生效、DNS 正确性 |
| IP（PING/HTTPS） | 选型阶段、裸机验线路、排查 DNS 问题 |

生产验收 **必须用域名 + HTTPS**，与用户真实访问一致。

### 1.3 你要的是「通」还是「快」？

| 指标 | 含义 | 工具 |
|------|------|------|
| 通畅率 | 多少节点能成功访问 | SpeedCE 通畅/异常数 |
| 延迟 | 成功节点的平均 ms | SpeedCE 平均延迟 |
| 页面性能 | LCP、FCP、TTFB | PageSpeed |

**先保证通，再优化快。** 地图大片红时，谈 PageSpeed 没意义。

## 第二章：SpeedCE 双视图快速上手

### 2.1 为什么推荐「双视图」工具

出海团队常见痛点：

- 用国内工具测，海外节点少；
- 用 Ping.pe 测，国内三网粒度粗；
- 两个网站来回切，截图对不上时间。

SpeedCE 在同一页面切换 **中国节点 / 全球节点**，HTTP、HTTPS、PING 协议一致，适合 **一份报告里放两张地图**。

### 2.2 标准操作流程

1. 打开 https://speedce.com/?lang=zh-CN
2. 输入域名，如 `www.yourproduct.com`
3. 选 **HTTPS**
4. 先测 **全球节点** → 保存地图与数据
5. 切换 **中国节点** → 再测 → 保存
6. 若国内业务重要，做电信/联通/移动三网分离

### 2.3 四个状态怎么读

| 状态 | 含义 |
|------|------|
| 通畅（绿） | 该节点 HTTPS 访问成功 |
| 异常（红） | 超时、连接失败、证书错误等 |
| 检测中 | 进行中 |
| 等待 | 排队 |

## 第三章：全球节点地图——分市场验收标准

以下延迟为 HTTPS 成功节点均值，仅供参考；**通畅率优先于延迟**。

### 3.1 北美（美国、加拿大）

**你关心的**：美国东海岸、西海岸节点是否绿。

| 指标 | 建议达标线 |
|------|------------|
| 通畅率 | ≥ 95% |
| 美国节点延迟 | 源站在美：< 80ms；源站在亚：150–250ms 可接受 |
| 异常分布 | 不应整州全红 |

**地图解读**：

- 美东绿、美西红：可能源站在弗吉尼亚，西海岸用户路由差 → 考虑全美 CDN
- 全美红：DNS、证书、或源站防火墙未放行海外

**机房建议**：用户主要在美 → 源站或 CDN 边缘在美；源站在亚洲则 **必须** 上全球 CDN。

---

### 3.2 欧洲（英、德、法、荷）

| 指标 | 建议达标线 |
|------|------------|
| 通畅率 | ≥ 95% |
| 欧洲节点延迟 | 源站在欧：< 50ms；源站在美：80–150ms |
| 合规 | GDPR、cookie -banner 与测速无关，但证书必须有效 |

**地图解读**：

- 英国绿、德国红：可能 CDN 未覆盖中欧节点
- 欧洲全绿但中国红：正常——源站在海外，国内访问慢是预期

---

### 3.3 亚太（新加坡、日本、香港、澳大利亚）

| 指标 | 建议达标线 |
|------|------------|
| 通畅率 | ≥ 95% |
| 东南亚延迟 | 源站在新：< 30ms；源站在美：150–220ms |
| 澳大利亚 | 常比东南亚高 30–50ms，单独看 |

**地图解读**：

- 新加坡绿、印尼红：CDN 东南亚覆盖不均
- 香港绿、澳洲红：澳新常需独立节点或 Anycast

---

### 3.4 中东、拉美、非洲（新兴市场）

这些区域节点数通常少于欧美，**个别红点不一定代表失败**——看通畅率趋势和是否持续。

**建议**：

- 业务重点在巴西 → 单独看南美节点
- 中东（阿联酋等）→ 看中东标注节点
- 通畅率 < 80% 且业务重点在该区 → 必须优化，不能忽略

## 第四章：中国节点地图——出海团队为什么还要看国内？

### 4.1 三类「必须看中国」的情况

1. **国内有研发/运营团队**，每天要登录后台
2. **双轨业务**：国内版 + 国际版
3. **备案与合规**：国内域名跳转、ICP 相关

### 4.2 源站在海外时，中国地图「慢」是否正常？

**通常正常。** 跨境链路长，中国电信/联通访问美国源站 150–250ms 很常见。

你要区分的是：

| 形态 | 含义 |
|------|------|
| 全国绿、延迟高 | 能访问，只是慢——可接受则用，不行则上国内 CDN 或国内镜像 |
| 大面积红 | 可能被墙、DNS 污染、或源站封了国内 IP——要专项排查 |
| 仅移动红 | 移动国际出口问题，考虑优化或接受 |

### 4.3 三网分离（国内团队必做）

中国节点下筛选 **电信 / 联通 / 移动**，各测一次。

团队若分散在北上深，运营商不同——**一张「全绿」地图可能掩盖移动极差**。

## 第五章：出海全生命周期测速清单

### 5.1 阶段一：选型（机房 / 云区域）

**目标**：决定源站放哪。

| 步骤 | 操作 |
|------|------|
| 1 | 用商家测试 IP，SpeedCE 全球节点 + PING/HTTPS |
| 2 | 对比美东、美西、新加坡、法兰克福延迟 |
| 3 | 中国节点测一遍——看团队能否接受管理延迟 |
| 4 | 截图存档，选型会议纪要附件 |

**决策表**：

| 主用户区 | 推荐源站区域 | 必须配套 |
|----------|--------------|----------|
| 美国 | 美东或美西 | 全美 CDN（若用户跨海岸） |
| 欧洲 | 法兰克福 / 伦敦 | 欧洲 CDN |
| 全球 | 美东 + 全球 CDN | Cloudflare 等 |
| 东南亚 | 新加坡 | 东南亚 CDN 节点 |
| 全球 + 中国团队 | 新加坡或美西 | 全球 CDN + 国内访问方案 |

---

### 5.2 阶段二：上线前验收

**可打印清单**：

```
全球节点 HTTPS：
  □ 美国通畅率 ≥ 95%
  □ 欧洲主要国家通畅率 ≥ 95%
  □ 目标新兴市场通畅率 ≥ 90%
  □ 无全球大面积证书错误（HTTPS 红、HTTP 绿 → 查证书）

中国节点 HTTPS（若有国内团队）：
  □ 通畅率 ≥ 85%（海外源站）或 ≥ 95%（国内源站）
  □ 电信/联通/移动无不可接受大片红
  □ 团队主要城市节点可访问

子域：
  □ www / api / app 分别测过
  □ 静态资源域（若独立）测过

DNS：
  □ 全球节点解析到预期 CDN/源站
  □ TTL 已按需调整

存档：
  □ 全球 + 中国地图截图已保存
```

### 5.3 阶段三：上线后第一周

| 时间 | 测什么 |
|------|--------|

## 第六章：八大出海故障场景

### 场景 1：全球绿、中国全红

**可能原因**：域名被墙、DNS 污染、国内防火墙拦截。

**SpeedCE 测法**：全球绿 + 中国红，高度怀疑合规/墙问题。

**处理**：SpeedCE 确认网络层；用 VSPING、BOCE 等做污染/墙检测；合规域名、备案、内容自查。

---

### 场景 2：中国绿、全球红

**可能原因**：

- 源站在国内，未备案域名海外访问受限
- 防火墙只放了国内 CDN
- GeoDNS 把海外解析到错误 IP

**处理**：检查 DNS 分线路解析；海外是否走 CDN；安全组是否全球可达。

---

### 场景 3：欧美绿、东南亚红

**可能原因**：CDN 东南亚节点弱或未部署。

**处理**：CDN 控制台开东南亚加速；或换覆盖更好的 CDN。

---

### 场景 4：主站绿、API 红

**可能原因**：`api.` 子域未配 CDN、证书漏签、Geo 分流错误。

**SpeedCE 测法**：分别测 `www` 和 `api`。

---

### 场景 5：间歇性全球 sporadic 红

**可能原因**：源站过载、DDoS、CDN 回源不稳定。

**处理**：连续测 6 次（每 15 分钟）；对照源站 IP；查 CDN 回源 5xx。

---

### 场景 6：延迟「能通但很慢」

**形态**：全球全绿，但均值 > 300ms。

**判断**：能通但体验差——需 CDN、压缩、边缘计算，不是换测速工具能解决的。

**分工**：SpeedCE 确认「通」；PageSpeed 优化「快」。

---

### 场景 7：IPv6 用户访问异常

**现象**：IPv4 地图全绿，部分 IPv6 用户反馈失败。

**SpeedCE 测法**：若有 IPv6 地址，分别用 IPv4、IPv6 目标测（SpeedCE 支持 IPv4/IPv6）。

---

### 场景 8：多语言站点路径问题

**说明**：SpeedCE 测的是 **域名层可达性**，不测 `/en/` `/de/` 路径。若主域绿但某语言 404，需应用层排查。

## 第七章：出海架构与测速对照

### 7.1 典型架构测什么

| 架构 | 测速目标 |
|------|----------|
| 单源站 + 全球 CDN | CDN 域名全球/中国地图 |
| 多区域源站（美 + 欧） | 各区域域名或 GeoDNS 解析结果 |
| 国内 + 海外双站点 | `.cn` 测中国，`.com` 测全球 |
| K8s 多集群 | 入口域名 + 各集群 Ingress |

**原则**：用户浏览器地址栏里输入什么，就测什么。

### 7.2 CDN 与源站对照（出海版）

| CDN 域名 | 源站 IP | 解读 |
|----------|---------|------|
| 全球绿 | 全球绿 | 理想 |
| 全球绿 | 全球红 | CDN 有效掩盖源站弱项（注意回源带宽） |
| 全球红 | 全球绿 | CDN 配置问题 |
| 全球红 | 全球红 | 源站或 DNS 全局故障 |

## 第十章：常见误区

### 误区 1：「Cloudflare 橙云开了就全球都快」

错。国内访问 Cloudflare 免费版可能很慢——中国地图必须单独看。

### 误区 2：「全球平均延迟 100ms 很好」

平均值掩盖问题。要看 **目标国家** 节点，不是全球平均。

### 误区 3：「海外用户不会用移动网络」

会。且 Wi-Fi 质量参差。通畅率比单次延迟重要。

### 误区 4：「测一次就上线」

至少测：全球白天（目标时区）、中国工作日、上线后 72h 复测。

### 误区 5：「网络没问题就是产品没问题」

SpeedCE 只测网络可达。登录失败、支付失败、CORS 错误——查应用日志。

## 结语：出海不怕慢，怕的是没有地图

全球化部署的复杂度，很大程度体现在 **地理分布** 上。你没法飞遍每个国家帮用户试，但可以用多节点测速，把他们的视角画在地图上。

**中国节点** 告诉你团队和国内市场能不能用；**全球节点** 告诉你卖不卖得了。两张图，一个工具，比一百句「我这边很快」靠谱。
